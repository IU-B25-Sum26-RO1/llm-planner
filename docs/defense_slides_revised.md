### Как снять за 10 минут

1. Запустить сцену: `ros2 launch ur10e_scene start_scene.launch.py` (или docker-compose `simulation`).
2. **Hero / GUI:** Win+Shift+S или скрин окна Gazebo → `01_scene_hero.png`, `06_gazebo_gui.png`.
3. **Последовательность:** `ros2 run ur10e_control_system cli pick <object>` затем `place <tray>` — три скрина mid-motion.
4. **Видео:** Win+G (Xbox Game Bar) или OBS, 1080p, 15–40 с одного успешного pick→place; обрезать тишину в начале.
5. Положить файлы в `docs/defense_assets/` с именами из таблицы.

В PowerPoint: **Вставка → Рисунки / Видео**. Для видео включить «воспроизводить при щелчке» и поставить постер = первый кадр.

## Слайд 2 · Проблема и мотивация

**Зачем это нужно**

- Манипуляционному роботу нужен естественный интерфейс: оператор говорит задачу — система переводит её в действия.
- Жёстко заданные сценарии плохо обобщаются; LLM выполняет семантический разбор команды в структурированный набор задач.
- Для команд вроде «возьми красный кубик» нужно связать описание с объектом на сцене. В проекте для этого заложена сегментация на основе SAM 3.

Карточки:
1. Естественный языковой интерфейс  
2. Семантический разбор команды (LLM → JSON)  
3. Восприятие объектов на основе SAM 3  

**Медиа:** нет.

## Слайд 3 · Цель и задачи

**Цель.** Построить систему «голос → исполнение задачи роботом» с LLM-семантическим парсером и симуляционным стендом UR10e.

1. LLM-декомпозер: естественный язык → JSON (actions: pick, place, go_home, open/close_gripper, stop, …).
2. Интеграция ASR (Vosk, ru): голос → текст → план.
3. Базовые политики манипуляции в Gazebo: pick / place / go_home (IK + ros2_control), координаты объектов из `/gazebo/model_states`.
4. Инфраструктура SAM 3 (bridge + preprocessor) как задел под зрительное обнаружение объектов.
5. Сквозная демонстрация: голосовая/текстовая команда → исполнение на стенде.

**Медиа:** нет.

## Слайд 6 · Реализация

- LLM-декомпозер: system prompt, JSON-схема задач/объектов/placement, ROS-нода `decomposer`.
- ASR: `audio_processor` на Vosk, публикация распознанного текста.
- Task Manager: последовательное исполнение задач через ROS 2 Action `/execute/base_action`.
- Robot Interface: pick / place / go_home / jog; координаты объектов из Gazebo model states; kinematic attach для удержания.
- Симуляционный стенд UR10e + камера, полная контейнеризация (docker-compose).
- SAM 3: preprocessor + WebSocket bridge к внешнему серверу сегментации (основа под vision-based grasp).

Боковой блок вместо «библиотека 5–8 навыков / grasp+place обучены»:

> **Рабочие политики:** pick, place, go_home, open/close_gripper, stop  
> Обучение IL/RL политик — не выполнялось  
> Верификация / replanning — не реализованы  

**Медиа:** `06_gazebo_gui.png`.

## Слайд 10 · Слабые места (актуализировать)

1. **Галлюцинация / невалидный JSON** — ловится частично (`json_utils`); pydantic-схема есть, но runtime-валидация слабая.  
2. **Нет vision в контуре управления** — координаты из Gazebo GT; SAM 3 пока не влияет на grasp.  
3. **Нет верификации и replanning** — при fail задачи оркестратор только логирует, план не перестраивается.  
4. **Ошибки ASR** — Vosk на шуме; рассинхрон топиков (`/recognized_text` vs `/recognized_textt`) ломает сквозной голос.  
5. **Расхождение schema ↔ prompt** — в prompt `pick`/`place`, в pydantic — `pick_object`/`place_object`.
